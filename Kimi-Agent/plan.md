# 常识Agent 数据库拓展计划

## 项目现状
- **当前总量**: 20条（4个类别，空间常识缺失）
- **目标**: 拓展至 **50条+**，覆盖 **7个类别**
- **Schema保持**: id, category, question, answer, trap, context, related, source, difficulty, tags

## 拓展策略

### Stage 1 — 补充缺失基础类别 + 现有类别扩容（并行收集）
按类别分派专门Agent并行收集：

| 类别 | 任务 | 新增数量 | Agent |
|------|------|----------|-------|
| **空间常识** | **从零补充**（地铁、楼层、空间距离等） | 5条 | 空间常识收集员 |
| 量化常识 | 扩容（温度感知、重量参照、面积大小等） | 3条 | 量化常识扩容员 |
| 流程常识 | 扩容（银行办卡、护照办理、离职交接等） | 3条 | 流程常识扩容员 |
| 社交语义 | 扩容（饭局座次、红包规则、送礼潜台词等） | 3条 | 社交语义扩容员 |
| 时间常识 | 扩容（保质期解读、工期估算、季节感知等） | 3条 | 时间常识扩容员 |

**新增3个全新类别**（每个5条）：

| 新类别 | 方向 | Agent |
|--------|------|-------|
| **消费常识** | 折扣计算、会员规则、价格陷阱等 | 消费常识收集员 |
| **数字生活常识** | APP权限、WiFi密码、扫码点餐等 | 数字生活收集员 |
| **生活避险常识** | 电梯故障、地震、诈骗识别等 | 避险常识收集员 |

### Stage 2 — 数据整合与质量审查
1. 合并所有Agent输出
2. 统一ID格式（category-XXX格式，补齐缺失编号）
3. 检查related字段关联完整性（新增数据与旧数据建立关联）
4. 验证JSON格式正确性
5. 确保总数≥50条，覆盖7个类别

### Stage 3 — 输出交付
1. 生成完整数据库 `/mnt/agents/output/commonsense_database.json`
2. 按类别拆分输出便于管理
3. 提供数据统计报告

## 数据质量要求（与原有保持一致）
- answer字段必须准确、具体、可验证
- trap字段必须解释"为什么容易搞混"
- 每条常识必须有明确的context场景
- difficulty评级合理（easy/medium/hard）
- tags至少3个，便于搜索匹配
- 尽量与已有数据建立related关联
